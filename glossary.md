## claude -p
category: スクリプト

CCがスクリプト（自動処理）の中に書く、Claudeへの一回きりの質問命令。

**【CCがこれを使うとき、何が起きているか】**
CCの回答にこれが出てきたとき、それは「自動で動くプログラム」を書いている場面です。`claude -p "〇〇を教えて"` という形で、プログラムが自動的にClaudeに質問して、返ってきた答えを受け取ります。あなたが画面を見ていなくても勝手に動きます。

**【挙動イメージ】**
上司が部下に「これ調べといて、結果だけ紙に書いて机に置いといて」と頼むようなイメージ。部下（Claude）は答えだけ返してそれで終わり。会話は続かない。

**【実例】**
- 記事のキーワードリストが100件あって、それを自動でカテゴリ分けするスクリプトを作るとき、CCが1件ずつ `claude -p "このKWは商標系？情報系？"` と投げて判定結果を受け取る
- CCがSlackに通知を送るスクリプトを書くとき、送る文章の日本語チェックを `claude -p "この文章おかしい箇所ある？"` に任せる

**【あなたが知っておくべきこと】**
- あなたが自分で入力するものではない。CCが自動処理の中に組み込むもの
- これが出てきたら「裏でCCが別のClaudeに質問している」と思えばOK

## config
category: CC設定

CCの動き方を決める「設定ファイル」のこと。

**【CCがこれを使うとき、何が起きているか】**
CCの回答に「config」が出てきたとき、それはCCの動き方を制御するファイル（`~/.claude/settings.json`）を読み書きしている場面です。言語設定・使うモデル・許可する操作の範囲など、CCの基本的なふるまいがここで決まります。

**【挙動イメージ】**
スマホの「設定」アプリと同じ。通知をONにしたり、言語を変えたりする場所。CCにとっての「設定アプリ」がconfigファイルです。

**【実例】**
- CCが「日本語で回答する」「sonnetモデルを使う」という設定で動いているのは、configに `"language": "日本語"` と書かれているから
- CCが新しい操作を許可するとき、configの `permissions` という項目を書き換えている

**【あなたが知っておくべきこと】**
- 場所は `~/.claude/settings.json`。直接開いて見ることもできる
- CCが「設定を変更します」と言ったときは、このファイルが書き換わる

## Dispatch
category: ツール

「今すぐ動け」という合図を別のシステムに送って処理を起動させる仕組み。

**【CCがこれを使うとき、何が起きているか】**
CCの回答に「Dispatch」が出てきたとき、それは「別のシステムやプログラムに対して開始の信号を送る」処理を書いている場面です。受け取った側は信号を受けて動き始めます。CCが何かを自動実行させたいとき、その「起動命令」がDispatchです。

**【挙動イメージ】**
工場の司令塔が各部署に「今すぐ出荷作業を開始しろ」と無線で指令を出す感じ。指令（Dispatch）を受けた部署（システム）が一斉に動き始める。

**【実例】**
- CCがGitHub Actionsの自動処理を手動で起動するとき、`workflow_dispatch` という信号を送る。これがDispatch
- 用語集のページを更新したとき「GitHub Pagesのビルドをdispatchした」という表現で出てくることがある

**【あなたが知っておくべきこと】**
- Dispatchは「起動命令を送った」という意味。実際の処理は受け取った側がやる

## -f
category: コマンドオプション

コマンドの後ろにつける「追加の指示」のひとつ。「強制的にやれ」「ファイルを指定する」など、コマンドによって意味が変わる。

**【CCがこれを使うとき、何が起きているか】**
CCの回答に `-f` が出てきたとき、それはコマンド（命令文）に「オプション」をつけている場面です。`-f` はコマンドによって意味が違います。たとえば `rm -f` なら「確認なしで強制削除」、`git push -f` なら「強制的に上書き送信」、`docker build -f` なら「使う設計図ファイルを指定」という意味です。`-` のあとにアルファベット1文字がつく形は「フラグ」や「オプション」と呼ばれ、コマンドの動きを細かく調整するものです。

**【挙動イメージ】**
料理の注文に「辛さ増しで」「大盛りで」とオプションをつけるのと同じ。`-f` は注文票に書き加える追加指示のひとつです。何の注文（コマンド）につけるかで意味が変わります。

**【実例】**
- CCが `git push -f origin main` と書いたとき → 「リモートの内容を無視して、こちらの内容で強制的に上書きする」という意味。これは他の人の変更を消す可能性がある危険な操作
- CCが `rm -f ファイル名` と書いたとき → 「本当に消していいですか？」の確認を飛ばして即座に削除するという意味
- CCが `docker build -f Dockerfile.prod .` と書いたとき → 「Dockerfile.prodという名前の設計図ファイルを使ってビルドしろ」という意味

**【あなたが知っておくべきこと】**
- `-f` は「force（強制）」の略であることが多い。強制＝確認を飛ばすということなので、データが消えるリスクがある
- CCが `-f` つきの削除コマンドやGit操作を提案してきたら、「本当にそれでいいか」を自分で判断してからOKを出すこと
- `-f` が「file（ファイル指定）」の略になるコマンドもある。文脈で判断する

## null
category: プログラミング基礎

「何もない」「値が入っていない」ことを表す特別な印。空っぽの箱のようなもの。

**【CCがこれを使うとき、何が起きているか】**
CCの回答に `null` が出てきたとき、それは「ここにはまだ何も入っていない」「この情報は存在しない」ということを表しています。たとえば設定ファイルやデータの中で、まだ値が決まっていない項目や、意図的に「なし」としている項目に `null` が使われます。0や空文字（""）とは違い、「そもそも値が存在しない」という状態です。

**【挙動イメージ】**
アンケート用紙で、まだ何も書いていない欄がある状態。空欄＝「回答なし」。これが `null`。「0点」と書いてあるのとは違います。0点は「0という回答がある」状態、空欄は「回答そのものがない」状態です。

**【実例】**
- CCが作ったJSONファイル（設定やデータを書く形式）で `"published_at": null` と書いてあったら → 「公開日がまだ決まっていない（未公開）」という意味
- CCが「この値は null です」と言ったら → 「その項目にはまだ何も設定されていません」という意味
- CCがエラーの原因を説明するとき「null を参照しようとしてエラーになった」→ 空っぽの箱の中身を取り出そうとして「中身がないよ」とコンピュータが怒った状態

**【あなたが知っておくべきこと】**
- `null` はバグの原因になりやすい。「値があるはず」の場所が `null` だとプログラムが止まることがある
- 設定ファイルで `null` になっている項目は「未設定」という意味なので、必要なら値を入れてあげる必要がある

## Bash
category: ツール

CCがあなたのパソコン上でコマンド（命令文）を実行するための窓口。CCの「手足」にあたる部分。

**【CCがこれを使うとき、何が起きているか】**
CCの回答で「Bashで実行します」「Bashツールを使います」と出てきたとき、それはCCがあなたのパソコンのターミナル（黒い画面）でコマンドを打ち込もうとしている場面です。ファイルの作成・削除、プログラムの実行、Gitの操作など、パソコンに対する「実際の操作」はほぼすべてこのBashを通して行われます。

**【挙動イメージ】**
CCの頭脳が「考える部分」だとすると、Bashは「手足」です。「このファイルを動かしたい」と考えたCCが、Bashという手を使って実際にパソコンを操作します。あなたが承認ボタンを押すまでは手が動かない（実行されない）仕組みです。

**【実例】**
- CCが `git push origin main` を実行しようとするとき → Bashを使ってGitHubにコードを送信している
- CCが `python スクリプト名.py` を実行しようとするとき → Bashを使ってプログラムを動かしている
- CCが `mkdir フォルダ名` を実行しようとするとき → Bashを使って新しいフォルダを作っている

**【あなたが知っておくべきこと】**
- Bashの実行は承認画面が出る。何をしようとしているか確認してからOKを押すこと
- 「Bash」はもともとLinux/Macで使われるコマンド入力システムの名前。CCではそれをツールとして使っている
- CCが使うツールの中で最も強力。ファイル削除や外部送信もできるので、承認時に内容を見る癖をつけるとよい

## enchanting
category: 英単語

「魅力的な」「うっとりするような」という意味の英語。技術用語ではない。

**【CCがこれを使うとき、何が起きているか】**
CC自身が使う用語ではなく、CCが読み込んだ英語の文章（GitHubのREADME、ライブラリの説明文、コメントなど）の中に含まれていることがあります。技術的な操作とは関係なく、単に英語の形容詞です。

**【挙動イメージ】**
日本語の記事に「素敵な」「魅力的な」と書いてあるのと同じ。飾りの言葉なので、読み飛ばしても意味は通じます。

**【実例】**
- GitHubのプロジェクト説明文に「an enchanting developer experience」と書いてあった → 「魅力的な開発体験」という意味。宣伝文句の一部
- CCが英語のドキュメントを要約するときに「enchanting」をそのまま残すことがある

**【あなたが知っておくべきこと】**
- 技術的な意味はない。「魅力的」と読み替えればOK

## og:image
category: SEO/Web

Webページを誰かに共有したとき、リンクのプレビューに表示される「代表画像」を指定するための設定。

**【CCがこれを使うとき、何が起きているか】**
CCがHTMLのheadタグ内に `<meta property="og:image" content="画像のURL">` を書いているとき、それは「このページをLINE・Slack・X（旧Twitter）などで共有したときに、どの画像をサムネイルとして表示するか」を指定しています。OGは「Open Graph」の略で、SNSやチャットアプリがリンクのプレビューを作るときに読む情報です。

**【挙動イメージ】**
YouTubeの動画にサムネイルを設定するのと同じ。動画一覧やリンク共有時に「この画像を見せてね」と指定しておくもの。og:imageはWebページ版のサムネイル設定です。

**【実例】**
- 記事をSlackに貼ったとき、記事のアイキャッチ画像がプレビューに出る → それはog:imageで指定された画像
- og:imageが設定されていないと、共有時にプレビューが真っ白になったり、意図しない画像が表示されることがある

**【あなたが知っておくべきこと】**
- WordPressでは通常、アイキャッチ画像がog:imageとして自動設定される。手動で設定する必要はほぼない
- og:image以外にも og:title（タイトル）、og:description（説明文）など、セットで使われる「OGタグ」がある

## fetch
category: プログラミング基礎

インターネット上のページやデータを「取りに行く」こと。英語で「取ってくる」という意味そのまま。

**【CCがこれを使うとき、何が起きているか】**
CCの回答に `fetch` が出てきたとき、それはインターネット上にあるWebページやデータ（APIの返答など）を取得しようとしている場面です。`fetch(https://~~~)` という書き方は、カッコ内のURLにアクセスして、そこにある情報を持ち帰る命令です。ブラウザでURLを開いて表示されるものを、プログラムが裏側で自動的に取りに行っている状態です。

**【挙動イメージ】**
コンビニにお使いを頼むのと同じ。「この住所（URL）のお店に行って、置いてある商品（データ）を持って帰ってきて」という指示がfetchです。お使い先が遠い（サーバーの応答が遅い）と時間がかかることもあります。

**【実例】**
- CCがWebページの内容を調べるとき、`fetch("https://example.com")` でそのページのHTMLを取得する。あなたがブラウザでそのページを開くのと同じことを、プログラムが裏で自動的にやっている
- CCがAPIからデータを取るとき、`fetch("https://api.example.com/data")` で情報を受け取る。たとえば天気予報APIから今日の天気を取得する、など
- JavaScriptのコードの中で `fetch` と書いてあったら、それはWebサイトが裏側でサーバーと通信している部分

**【あなたが知っておくべきこと】**
- fetchは「外部と通信する」操作なので、アクセス先のURLに個人情報や秘密の情報が含まれていないか注意すること
- fetchの結果は相手のサーバー次第。サーバーが落ちていたり、アクセス制限がかかっていると失敗する
- CCが使う「WebFetch」ツールもこれと同じ仕組み。URLを指定して、そのページの内容を読み取っている

## robots metaタグ
category: SEO/Web

Webページに埋め込む「検索エンジンへの指示書」。このページを検索結果に出してよいか、リンク先をたどってよいかなどをGoogleに伝える設定。

**【CCがこれを使うとき、何が起きているか】**
CCの回答に「robots metaタグ」が出てきたとき、それはWebページのHTMLの先頭部分（headタグ内）に `<meta name="robots" content="noindex, nofollow">` のような1行を書いている、または確認している場面です。この1行で「Googleの検索結果にこのページを載せるか載せないか」「このページ内のリンク先をGoogleがたどるかたどらないか」を指定しています。

**【挙動イメージ】**
お店の入口に「関係者以外立入禁止」の看板を出すようなもの。看板（robots metaタグ）を見たGoogleの巡回ロボットが「このお店（ページ）は入らないでおこう」と判断します。看板がなければ自由に入って中を見て回ります（＝検索結果に載る）。

**【実例】**
- CCがSEO記事のHTMLを確認しているとき「robots metaタグが noindex になっています」と言ったら → そのページはGoogleの検索結果に表示されない設定になっている、という意味
- CCがWordPressの記事を下書き（draft）で入稿するとき、下書き状態のページにはWordPress側で自動的に noindex が付くことがある → 公開すれば外れる
- サイト全体ではなく特定のページだけ検索結果から除外したいとき、そのページのHTMLに `<meta name="robots" content="noindex">` を入れる

**【あなたが知っておくべきこと】**
- 間違えて公開ページに `noindex` をつけると、Googleの検索結果から消えてアクセスが激減する。SEO記事では特に注意が必要
- よく使われる値は `index`（検索結果に載せてよい）、`noindex`（載せないで）、`follow`（リンク先をたどってよい）、`nofollow`（たどらないで）の組み合わせ
- robots metaタグとは別に `robots.txt` というファイルもある。robots.txtはサイト全体への指示、robots metaタグは1ページごとの指示、という違いがある

## バリデーション
category: プログラミング基礎

入力された値が「ルール通りかどうか」をチェックする仕組み。門番のような役割。

**【CCがこれを使うとき、何が起きているか】**
CCの回答に「バリデーション」が出てきたとき、それは「データや入力内容が正しい形式・範囲に収まっているか確認する処理」について話しています。たとえばメールアドレス欄に電話番号が入っていないか、数字を入れるべき欄に文字が入っていないか、といったチェックのことです。CCがコードを書くときに「バリデーションを追加する」と言ったら、「おかしなデータが入ってきたときに弾く仕組みを作る」という意味です。

**【挙動イメージ】**
遊園地の入口で身長制限をチェックするスタッフのようなもの。「120cm以上ですか？」と確認して、OKならアトラクションに乗れる、NGなら「乗れません」と止める。バリデーションはこの「入口でのチェック」にあたります。

**【実例】**
- CCがフォームを作るとき「メールアドレスのバリデーションを入れます」と言ったら → 入力欄に `@` が含まれているか、形式が正しいかをチェックする処理を追加するという意味
- CCが設定ファイルを読み込むコードを書くとき「値のバリデーションをします」と言ったら → 必須項目が空になっていないか、数字が想定の範囲内かなどを確認する処理のこと
- CCがAPIにデータを送る前に「バリデーション通ってから送信します」と言ったら → おかしなデータを外部に送らないよう、事前に内容を検品しているという意味

**【あなたが知っておくべきこと】**
- バリデーションが「ない」と、間違ったデータがそのまま保存されたり送信されたりしてトラブルの原因になる
- 「バリデーションエラー」と表示されたら、入力した内容がルールに合っていないという意味。何がダメだったかエラーメッセージを確認すればよい
